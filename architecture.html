Architecture

System Design

A layered architecture where every life event flows through carefully orchestrated data structures — from input to persistent memory.

System Flow

How Data Flows Through Digital Shadow

Every user interaction passes through six interconnected layers, each powered by purpose-built C data structures.

👤
Layer 1 — Input
User
The human operator interacting with the Digital Shadow system via keyboard commands and menu selections.
🖥️
Layer 2 — Interface
Console Interface (UI)
A menu-driven console application that captures user input, validates data, and dispatches commands to the appropriate subsystem.
🔗
Layer 3 — Storage
Linked List (Event Storage)
A singly linked list that dynamically stores every life event. Nodes are allocated with malloc() and linked via pointers, enabling efficient insertion and deletion.
📚
Layer 4 — Undo & Replay
Stack (Undo / Replay)
A LIFO stack that records operations for undo capability and enables reverse-chronological replay of daily events using push and pop operations.
📊
Layer 5 — Processing
Analytics (Statistics Engine)
A single-pass aggregation engine that traverses the linked list to compute statistics: total events, events per day, category breakdown, and timeline density.
📋
Layer 6 — Presentation
Output (Timeline Display)
Formatted console output that renders the event timeline chronologically, presenting dates, times, event descriptions, and categories in a readable format.
Components

Component Deep Dive

Each architectural layer is purpose-built with specific responsibilities, data structures, and algorithmic guarantees.

👤

User Layer

The entry point of the system. The user interacts through a numbered menu, providing event details like description, date, time, and category.

  • Keyboard-driven input via scanf() and fgets()
  • Menu choices routed via switch-case dispatcher
  • Input validation prevents malformed data entry
  • Error messages guide correction of invalid input
🖥️

Console Interface

The UI layer presents a clear menu of 10 operations and handles all I/O formatting. It acts as the bridge between human intention and machine execution.

  • 10-option main menu with numbered selection
  • Structured prompts for event data collection
  • Color-coded output using ANSI escape sequences
  • Confirmation prompts before destructive operations
🔗

Linked List Storage

The core data structure. Each node holds one life event and a pointer to the next. The list grows and shrinks dynamically using heap memory allocation.

  • Singly linked list with head pointer
  • O(1) insertion at head, O(n) at tail
  • Dynamic memory via malloc() and free()
  • Linear traversal for search, display, and statistics
  • No fixed capacity — grows with user activity
📚

Stack (Undo / Replay)

The stack provides LIFO ordering for two critical features: undoing the last operation and replaying a day's events in reverse chronological order.

  • Array-based stack with top pointer
  • O(1) push and pop operations
  • Replay: push all day events, then pop to reverse
  • Undo: pop last action and reverse its effect on the list
  • Overflow/underflow protection with clear messages
📊

Analytics Engine

A stateless computation module that performs single-pass analysis over the linked list to extract meaningful statistics about the user's digital life.

  • Total event count computed in O(n)
  • Events grouped by date for daily density
  • Category frequency breakdown
  • Most active day and time range identification
  • No additional data structures — aggregates in-place
📋

Timeline Display

The presentation layer formats raw linked list data into a human-readable timeline, grouping events by date and displaying them with clear formatting.

  • Chronological display with date headers
  • Formatted output: [Time] Event — Category
  • Today's view filters by current system date
  • Empty state messaging when no events exist
  • Replay view shows reverse-order playback

Memory Lifecycle

Digital Shadow carefully manages heap memory — every node that is created is eventually freed, preventing memory leaks.

🟢

Allocation

When an event is added, the system calls malloc(sizeof(struct Event)) to allocate a block of memory on the heap. The returned pointer becomes the address of the new node.

  • Each node occupies ~128 bytes (struct size)
  • Pointer validity checked (NULL = allocation failure)
  • Node fields populated before linking into list
🔗

Linking

The new node's next pointer is set to NULL, and the previous tail's next is updated to point to this new node. The list grows by one element.

  • Head insertion: new→next = head; head = new
  • Tail insertion: traverse to end, tail→next = new
  • Pointer reassignment is O(1) after reaching position
🔴

Deallocation

When an event is deleted or the program exits, free() returns the memory to the operating system. The pointer is set to NULL to prevent dangling references.

  • Predecessor's next updated before free()
  • Exit routine walks entire list, freeing every node
  • Stack memory freed separately from list memory
Principles

Design Principles

The architecture of Digital Shadow follows software engineering principles adapted for systems-level C programming.

🏗️

Separation of Concerns

Each module handles one responsibility. The UI knows nothing about memory management. The linked list has no knowledge of console formatting. The stack operates independently of analytics.

🧩

Modular Design

Functions are grouped by purpose into logical modules: event operations, stack operations, statistics computation, and display formatting. Each can be tested and modified independently.

💾

Manual Memory Management

Without garbage collection, Digital Shadow explicitly manages every byte of heap memory using malloc() and free(), ensuring zero memory leaks through disciplined pointer management.

🛡️

Defensive Programming

Every pointer is checked before dereference. Every allocation is verified. Every user input is validated. Edge cases like empty lists and full stacks are handled gracefully.
